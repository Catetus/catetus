# SPEC-0001 — Canonical Splat Internal Representation (`SplatIR`)

**Status:** Implemented (Phase 1)
**Crate:** `catetus-core`

## Goal

Define a deterministic canonical splat representation used by all importers, optimizers, exporters, and tests. Independent of PLY, SPZ, glTF, or OpenUSD.

## Data model

```rust
pub struct Splat {
    pub position: [f32; 3],
    pub rotation: [f32; 4], // unit quaternion (x, y, z, w)
    pub scale:    [f32; 3], // log-scale on disk, linear in IR
    pub opacity:  f32,
    pub color:    Color,    // RGB or SH coefficients
}

pub enum Color {
    Rgb([f32; 3]),
    Sh { degree: u8, coeffs: Vec<f32> },
}

pub struct SplatScene {
    pub splats: Vec<Splat>,
    pub coordinate_system: CoordinateSystem,
    pub semantic_labels: Option<Vec<SemanticLabel>>,
    pub temporal_mode: TemporalMode, // Static for v1
}
```

## Requirements

* Deterministic ordering (input order preserved unless explicitly sorted)
* Lossless mode for round-trip parity
* Explicit attribute availability flags
* Coordinate-system metadata (Y-up vs Z-up, right- vs left-handed)
* Stable BLAKE3 hash of the canonical byte representation
* Streamable iteration via `Iterator<Item = &Splat>`
* Reserved temporal/4D fields (`TemporalMode::Static` for v1)
* Optional semantic labels

## Acceptance tests

```gherkin
Feature: Canonical IR

Scenario: Load tiny synthetic scene into IR
  Given a synthetic scene with 3 splats
  When I serialize it into SplatIR
  Then the IR contains 3 splats
  And every splat has position, rotation, scale, opacity, and color
  And the scene hash equals the golden hash

Scenario: IR serialization is deterministic
  Given the same input parsed twice
  When I serialize both IR outputs
  Then the byte outputs are identical
  And the hashes are identical

Scenario: IR reserves temporal fields
  Given a static splat scene
  When I parse it into SplatIR
  Then temporal mode is "static"
  And no dynamic frame data is required
```
